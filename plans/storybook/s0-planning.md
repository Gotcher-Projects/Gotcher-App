## Lulu API — Print-on-Demand Evaluation

### What Lulu does
Lulu (lulu.com) is a print-on-demand service with a REST API. You upload a formatted interior
PDF and a cover PDF; Lulu prints and ships a physical book directly to the customer. No
inventory, no fulfillment — you pay per-unit at order time.

### Why it fits
- The app already generates PDFs (`Frontend/src/lib/pdf.js` + jsPDF). Lulu just needs a
  properly formatted interior PDF handed to it.
- It's a natural premium upsell on top of the existing paid storybook tier: digital book is
  the base paid feature; physical print is a one-time add-on purchase.
- No AI content work — Lulu is purely the output channel. Chapter text is already generated
  by Claude in S2.

### Rough economics
| Item | Estimate |
|------|----------|
| Lulu print cost (6×9 softcover, ~40 pages) | ~$8–12 |
| Suggested retail to user | $30–45 |
| Gross margin per book | ~$20–35 |

### Integration sketch
1. User clicks "Order a Printed Book" in the storybook view (new button, paid users only)
2. Frontend calls a backend endpoint that assembles the chapter PDF
3. Backend uploads PDF to Lulu via their API, gets back a print job ID and a checkout URL
4. User is redirected to Lulu's hosted checkout to pay and enter shipping address
5. Lulu handles payment + fulfillment — no Stripe integration needed on our side
6. (Optional) Webhook from Lulu notifies us when the order ships

### Key unknowns to resolve before committing to S5
- [ ] Does Lulu's API allow redirect-to-checkout flow, or does it require us to collect
      payment ourselves and then POST an order? (Check Lulu developer docs)
- [ ] What PDF spec does Lulu require? (trim size, bleed, embedded fonts, color profile)
      jsPDF may need to be replaced or wrapped for print-quality output
- [ ] Minimum page count for Lulu? A user with only 2 chapters may not meet the minimum
- [ ] Do we want to own the payment (more margin, more complexity) or send users to Lulu
      checkout (simpler, less margin)?
- [ ] Does Lulu support white-labeling so the package doesn't arrive branded "Lulu"?
- [ ] Review Lulu's API terms — any restrictions on reselling or subscription-model apps?

### Alternatives to evaluate
| Option | Notes |
|--------|-------|
| **Lulu API** | Most developer-friendly POD API; broad format support |
| **Blurb API** | Better print quality reputation; smaller developer community |
| **Printful** | Not book-focused; better for merch |
| **Peecho** | Europe-based; good for EU users |
| **Skip entirely** | PDF export + shareable link may be enough for V1; revisit after launch |

## Decisions to Make This Session

1. **Is S5 (print) in scope for the initial storybook launch, or a future addition?**
   Recommendation: defer to post-launch. S1–S4 deliver real value without the complexity.
   Revisit once actual users confirm demand.

2. **Pricing model for the digital storybook**
   - One-time purchase (flip `is_paid` after checkout)?
   - Monthly subscription?
   - Bundle with a future "family plan"?
   S1 implements the flag; the payment processor is wired in later. Decide the model now so
   the upgrade prompt copy (in `PaidGate`) is accurate.

3. **Chapter generation limits for free users**
   Should free users be able to see chapter titles/teasers but not the full text?
   Or is the entire storybook tab hidden until paid? Current plan (S1) hides everything.

4. **Cover design for the physical book (if S5 proceeds)**
   Lulu needs a cover PDF. Options: fixed template with baby's name overlaid, let user
   upload a photo, or offer a selection of illustrated covers. Scope this before building.
