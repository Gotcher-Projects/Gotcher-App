First-Time Deployment

Follow these steps in order the very first time you're standing up the app on the VPS.

1
Prerequisites — check these before touching the server
  • SSH access works: ssh deploy@87.99.153.7
  • The repo is pushed to GitHub (or your remote of choice)
  • DNS A records for @ and www point to 87.99.153.7
Cloudflare proxy must be OFF. The DNS record must show a grey cloud icon, not orange. Caddy provisions its own Let's Encrypt TLS certificate directly — Cloudflare's proxy intercepts that handshake and breaks HTTPS setup.

Verify DNS has propagated before continuing:

# Both should return 87.99.153.7
dig +short michaelgotcher.com
dig +short www.michaelgotcher.com
2
SSH into the VPS and clone the repo
ssh deploy@87.99.153.7

mkdir -p ~/gotcherapp
cd ~/gotcherapp

# Replace with the actual repo URL
git clone https://github.com/YOUR_USER/YOUR_REPO.git .
3
Create the .env file

The .env file holds all secrets and is never committed to the repo. You create it once manually on the VPS from the provided template. 

cd ~/gotcherapp
cp .env.prod.example .env
nano .env

Fill in every value. Use the commands below to generate the secrets:

# Generate JWT_SECRET (needs 32+ chars)
openssl rand -base64 48

# Generate DB_PASSWORD
openssl rand -base64 32

The completed .env should look like this:

APP_DOMAIN=michaelgotcher.com

JWT_SECRET=<your generated value>
DB_PASSWORD=<your generated value>
SECURE_COOKIES=true

# Optional — leave blank to disable photo uploads
CLOUDINARY_CLOUD_NAME=
CLOUDINARY_API_KEY=
CLOUDINARY_API_SECRET=

# Optional — leave blank to disable email features
SMTP_HOST=
SMTP_PORT=587
SMTP_USERNAME=
SMTP_PASSWORD=
4
Build and start all services
cd ~/gotcherapp
docker compose -f docker-compose.prod.yml up -d --build

This single command does everything:

  • Builds the frontend — Node 20 runs Vite, static files are baked into the Caddy image
  • Builds the API — Gradle compiles the fat JAR inside an Eclipse Temurin container
  • Pulls Postgres 16
  • Starts all three containers
  • Caddy automatically provisions a Let's Encrypt TLS certificate

Watch the logs until all services settle:

docker compose -f docker-compose.prod.yml logs -f

You're looking for these signals in the log output:

Service Expected log message
caddy serving initial configuration — no TLS errors
api Started GotcherAppApplication
postgres database system is ready to accept connections
The first build takes several minutes — Gradle downloads dependencies and compiles Java. Subsequent builds are much faster due to Docker layer caching.
5
Verify database migrations ran

Flyway runs all migrations automatically when the API starts. Confirm every table was created: 

docker compose -f docker-compose.prod.yml exec postgres \
  psql -U gotcherapp_app -d gotcherapp -c "\dt"

You should see:

users  |  refresh_tokens  |  baby_profiles  |  milestones
vaccine_records  |  appointments  |  journal_entries  |  first_times
growth_records  |  sleep_logs  |  feeding_logs  |  diaper_logs
flyway_schema_history

If a table is missing, check the API logs for Flyway errors: docker compose -f docker-compose.prod.yml logs api | grep -i flyway

6
Smoke test in the browser

Open https://michaelgotcher.com and work through this checklist:

  • HTTPS certificate is valid (green padlock, no browser warnings)
  • www.michaelgotcher.com redirects to the root domain
  • Register a new account
  • Login / logout / token refresh works
  • Save a baby profile (Dashboard tab)
  • Log a feeding, sleep, and diaper entry (Track tab)
  • Add a growth measurement — chart renders correctly
  • Toggle a vaccine (Health tab)
  • Add an appointment (Health tab)
  • Check a milestone (Health tab)
  • Add a journal entry with a photo (Memories tab)
  • Add a First Time with a photo (Memories tab)
  • PDF export downloads and opens correctly
  • Page reload preserves the logged-in session
  • Session expiry logs the user out cleanly

Updating the App (Code Changes)

After the first deployment, pushing code updates is a single script. Database migrations — if any were added — are applied automatically when the API restarts.

→
Run deploy.sh

Option A — directly on the VPS:

ssh deploy@87.99.153.7
cd ~/gotcherapp
./deploy.sh

Option B — one-liner from your local machine:

ssh deploy@87.99.153.7 "cd ~/gotcherapp && ./deploy.sh"

deploy.sh does three things:

  • git pull origin main — fetches the latest code
  • docker compose build — rebuilds only the images that changed
  • docker compose up -d — replaces running containers; unaffected services stay up
✓
Post-deploy sanity check
# Tail the last 50 log lines across all services
ssh deploy@87.99.153.7 "cd ~/gotcherapp && \
  docker compose -f docker-compose.prod.yml logs --tail=50"
  • API started without errors
  • Flyway printed migration messages or "Schema is up to date"
  • No container shows "Restarting" in docker compose ps

Database Backup & Restore

↓
Take a backup

Run this before any risky migration or when you want a point-in-time snapshot:

ssh deploy@87.99.153.7
cd ~/gotcherapp
./backup.sh
# Creates: backup_20260417_120000.sql.gz

Copy it to your local machine:

# Run locally:
scp deploy@87.99.153.7:~/gotcherapp/backup_*.sql.gz ./backups/

Restore from backup

Warning: This overwrites all current data in the database with the contents of the backup file. 
gunzip -c backup_20260417_120000.sql.gz | \
  docker compose -f docker-compose.prod.yml exec -T postgres \
  psql -U gotcherapp_app gotcherapp

Container Management

Check status of all containers

docker compose -f docker-compose.prod.yml ps

Stream live logs (all services)

docker compose -f docker-compose.prod.yml logs -f

Logs for a specific service (caddy, api, or postgres)

docker compose -f docker-compose.prod.yml logs -f api

Restart a single service without rebuilding

docker compose -f docker-compose.prod.yml restart api

Stop everything — data volume is preserved

docker compose -f docker-compose.prod.yml down

Stop and wipe the database — ALL DATA LOST

docker compose -f docker-compose.prod.yml down -v

Changing the Domain

⚙
All domain references flow from one variable
  1. Update APP_DOMAIN in .env on the VPS
  2. Update the DNS A records at your registrar to point to 87.99.153.7
  3. Run ./deploy.sh

Caddy will automatically provision a new Let's Encrypt certificate for the new domain. No manual cert management needed.

Rotating Secrets

⚙
JWT_SECRET and DB_PASSWORD

JWT_SECRET

Update the value in .env, then redeploy. All existing user sessions will be invalidated — everyone will need to log in again.

# Generate a new secret
openssl rand -base64 48

# Edit .env, paste the new value, save, then:
./deploy.sh

DB_PASSWORD

You must update both the .env file and the password inside Postgres — they must stay in sync.

# 1. Update the password inside the running Postgres container
docker compose -f docker-compose.prod.yml exec postgres \
  psql -U gotcherapp_app -d gotcherapp -c \
  "ALTER USER gotcherapp_app PASSWORD 'your-new-password';"

# 2. Update .env with the same new password, then redeploy
./deploy.sh
