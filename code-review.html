Executive Summary

The GotcherApp codebase is well-structured and demonstrates thoughtful engineering in its core patterns: ownership is checked via baby_profile_id in every backend service, JdbcTemplate queries use parameterized placeholders throughout (no SQL injection risk), JWT rotation is correctly implemented, and CORS is locked to a single configured origin. The frontend–backend split is clean, and the component decomposition in the React layer is reasonable for the current feature set.

The most urgent concerns are a handful of logic bugs in the frontend: sleep entries logged across midnight silently produce negative durations, the quick-log "Sleep" panel on the Dashboard fails when start and end times straddle midnight, and the months approximation formula (week / 4.345) drifts from the accurate calendar-month calculation that formatBabyAge already performs. On the backend, the seed script registers users with the key displayName instead of the correct display_name, so all seed-generated display names are silently dropped.

Test coverage is healthy for auth, feeding, milestone, and journal domains. Session 10 added tests for babyAge.js, utils.js, and extracted dashboard stat helpers to dashboardStats.js with full test coverage. Several newer modules still lack coverage — appointments, poop, sleep, first times, and the frontend PDF/image utilities. The highest-risk untested path is the token refresh race-condition de-duplication logic in api.js and the PDF generation pipeline in pdf.js.

Refactoring opportunities are mostly cosmetic: repeated ownership-resolution boilerplate across seven service classes could be unified, and several inline utility functions (date formatting, duration formatting) are copy-pasted across components. Overall the codebase is deployable and the foundations are solid; the P1 fixes are small and self-contained.

openCropModal leaks a React root if the parent unmounts before the user interacts
Frontend/src/lib/imageUtils.jsx · lines 135–152
openCropModal imperatively creates a new ReactDOM.createRoot outside the main React tree and appends it to document.body. P3
formatBabyAge can show "0 months and 0 days old" for a baby born today
Frontend/src/lib/babyAge.js · lines 22–32
When a baby is born today (totalDays = 0, weeks = 0), the code falls through to the weeks branch returning "0 weeks old". This is handled. However, the month branch sets months = 0 and days = 0 for babies 0–12 weeks and would return "0 months and 0 days old" if the weeks branch were removed or the condition changed. More practically, a baby exactly N months old with the anniversary calculation returns "N months old" — which is correct but misses that it only activates at 13+ weeks; the transition could be jarring (from "12 weeks old" to "3 months old" mid-day).
Fix (cosmetic): Add a unit test covering the exact 13-week boundary to lock in the expected display string. Consider adding "and 0 days" suppression in the month path as well (currently it returns "N months old" when days = 0, which is correct).

Missing Test Coverage

P1
No backend tests for appointments, sleep, poop, first-times, vaccines, or growth domains
Backend/src/test/java/com/gotcherapp/api/ — missing directories
Seven backend service/controller pairs (AppointmentService, SleepService, PoopService, FirstTimeService, VaccineService, GrowthService, BabyProfileService) have zero test coverage. These handle the bulk of the app's data mutations. The highest risk untested paths are ownership verification (baby_profile_id check preventing cross-user data access) and input validation (type allow-lists in PoopService and SleepService).
Fix: Write service unit tests (Mockito, like the existing FeedingServiceTest pattern) for each domain. Prioritize AppointmentService (has date validation), PoopService (type/color/consistency allow-lists), and FirstTimeService (label/date required fields).
P1
Token refresh race-condition deduplication in api.js has no test for concurrent 401s
Frontend/src/test/api.test.js — missing concurrent refresh test
The refreshing module-level Promise in api.js is meant to deduplicate simultaneous refresh calls from concurrent 401 responses. The existing tests only exercise single sequential refresh flows. There is no test that fires two simultaneous apiRequest calls, both receiving 401, and verifies that /auth/refresh is called exactly once.
Fix: Add a test that fires two apiRequest calls with Promise.all, sets fetch to return 401 for both, mocks one successful refresh response, and asserts fetch was called exactly 3 times (2 initial + 1 refresh + 2 retries = 5 total) — or confirms the deduplication behaviour directly.
P1
No tests for PDF generation (pdf.js) — complex logic with no coverage
Frontend/src/lib/pdf.js — no test file exists
generatePdf contains complex page-packing logic (countPages, entryBlockH, multi-image layout) that is entirely untested. Regressions in the PDF layout are invisible until a user actually exports a book. The loadImageDataUrl canvas conversion and resolveImgLayout portrait vs landscape branching are also untested.
Fix: Write unit tests for the pure helper functions (entryBlockH, resolveImgLayout, buildChartData in HealthTab, toImperial in growthPercentiles). For generatePdf, use a lightweight integration test with a JSDOM mock for canvas and jsPDF spies.

✓ RESOLVED (S10) — Tests added for babyAge.js utility functions

S10 added babyAge.test.js covering getWeek (null/empty → 0, future → 0, today → 0, known dates) and formatBabyAge (null, under-13-weeks, exact weeks, months, months+days). All dates hardcoded to avoid timezone drift.

P2
No tests for validateSession in auth.js (silent refresh path)
Frontend/src/test/auth.test.js — missing validateSession tests
validateSession has three branches: /auth/me succeeds, /auth/me returns non-401 error, and /auth/me returns 401 followed by a refresh attempt. The refresh sub-path (clears localStorage on failure, returns false) is entirely untested. This is the code that runs on every app boot.
Fix: Add three tests to auth.test.js covering: (1) me returns 200 → returns true; (2) me returns 401, refresh succeeds → returns true; (3) me returns 401, refresh fails → clears localStorage and returns false.
P2
EmailVerificationService.verify() has no tests — token replay, expiry, and success paths untested
Backend/src/test/java/com/gotcherapp/api/auth/ — missing EmailVerificationServiceTest
The email verification flow (token lookup, expiry check, user update, token deletion) is entirely untested. An expired-token path that fails to delete the token, or a race condition where two verify calls hit concurrently, would be invisible.
Fix: Add EmailVerificationServiceTest with Mockito mocks for the three paths: token not found → false; token expired → deletes token, returns false; valid token → updates user, deletes token, returns true.
P2
JournalService.update() dynamic SET clause builder has no tests
Backend/src/test/java/com/gotcherapp/api/journal/ — JournalServiceTest exists but may not cover partial update
The dynamic query builder in JournalService.update() (and mirrored in FirstTimeService.update() and AppointmentService.update()) produces different SQL depending on which fields are non-null. The case where setClauses.isEmpty() (no fields to update) returns the current row is a subtle path that should be explicitly tested.
Fix: Add test cases for the empty-patch path (all fields null → returns current row unchanged) and a partial-patch path (only title non-null → only title SET clause emitted). Similar tests are needed for FirstTimeService and AppointmentService.

Refactoring Opportunities

P2
Ownership resolution boilerplate (findProfileIdByUserId + empty check) duplicated across 7 service classes
Backend — FeedingService, SleepService, PoopService, GrowthService, JournalService, AppointmentService, FirstTimeService, VaccineService, MilestoneService
Every service method starts with the same two lines: Optional<Long> profileId = babyProfileRepository.findProfileIdByUserId(userId); if (profileId.isEmpty()) return List.of() / false / Optional.empty() / throw …. This pattern is copy-pasted ~30 times across the codebase with minor variations in the "missing" return value. Adding a new service requires remembering to add both lines in every method.
Fix: Add two helper methods to BabyProfileRepository: getProfileIdOrEmpty(userId) (returns Optional) and requireProfileId(userId) (throws if empty). Service methods can then use a one-liner. Alternatively, extract a ProfileContext base class or utility.
P2
formatDuration duplicated across TrackTab and HealthTab (DashboardTab version resolved in S10)
Frontend/src/components/tabs/TrackTab.jsx · lines 52–59, 380–386; HealthTab.jsx (inline)
S10 extracted fmtDuration and timeAgo from DashboardTab into dashboardStats.js. However, formatDuration / duration formatting still appears as separate inline definitions in TrackTab and HealthTab. If the formatting style needs to change, each remaining copy must be updated independently.
Fix: Move the remaining duration helpers into Frontend/src/lib/formatting.js (which currently only has formatEntryDate). Export formatDuration(seconds) from there and import in TrackTab and HealthTab. Consider also re-exporting fmtDuration/timeAgo from formatting.js for a single source of truth.
P2
Optimistic-delete + re-fetch-on-error pattern copy-pasted across 8 handlers in BabySteps.jsx
Frontend/src/components/BabySteps.jsx · lines 183–188, 210–215, 237–242, 250–255, 268–273, 299–304, 319–324
Every delete handler follows the same pattern: optimistic removal from state → API call → on catch, re-fetch from API to restore. This is 7 nearly identical blocks. When the bug fix for user-visible error feedback is applied (P1 finding above), it must be applied to all 7 places.
Fix: Extract a generic optimisticDelete(setState, id, apiCall, refetch) helper, or at minimum a shared deleteWithRecovery pattern. This also makes the P1 fix above a single change.
P2
MilestoneService and VaccineService are structural duplicates of each other
Backend — MilestoneService.java vs VaccineService.java (identical structure)
The two files differ only in table name (milestones vs vaccine_records), column name (milestone_key vs vaccine_key), and the error messages. Everything else — constructor, getKeys, achieve, unachieve — is structurally identical. Adding a new "checkbox-style" resource would require a third copy.
Fix: Extract a generic KeyedRecordService(tableName, keyColumn) base class or utility method that both services delegate to. Both service classes become thin wrappers configuring the table/column names.
P3
BabySteps.jsx local save/load helpers duplicate auth.js saveSession/getStoredSession pattern
Frontend/src/components/BabySteps.jsx · lines 14–23
The save(key, data) and load(key) functions in BabySteps.jsx are a generic localStorage wrapper, but auth.js already has saveSession / getStoredSession doing the same with a fixed key. These could be consolidated into a shared localStore.js utility.
Fix: Extract localStore.get(key) / localStore.set(key, data) into Frontend/src/lib/localStore.js and use it in both places.
P3
HealthTab.jsx is 669 lines — five logically separate components in one file
Frontend/src/components/tabs/HealthTab.jsx
HealthTab.jsx contains GrowthTab, VaccineTab, MilestonesTab, AppointmentTab, AppointmentRow, MetricChart, and several helper functions. This makes it difficult to navigate and test individual sub-components. TrackTab.jsx (762 lines) has the same issue with three sub-trackers.
Fix: Split into tabs/health/GrowthTab.jsx, VaccineTab.jsx, AppointmentTab.jsx, MilestonesTab.jsx, keeping HealthTab.jsx as the thin orchestrator. Similarly split TrackTab.jsx.
P3
mapRow() helper duplicated across every Service class with minor variations
Backend — JournalService, FeedingService, SleepService, PoopService, GrowthService, AppointmentService, FirstTimeService
Each service has a private mapRow(Map<String, Object> row) that casts database column values to Java types. The ((Number) row.get("id")).longValue() and toBigDecimal patterns repeat verbatim. If the DB driver changes or a new column type is introduced, each copy must be updated.
Fix: Extract a RowMapper utility class with static helpers like getLong(row, col), getString(row, col), getDecimal(row, col), and getTimestamp(row, col). Each service mapRow becomes much more readable.
P3
Confirm-delete UI pattern (Yes/No buttons appearing inline) is copy-pasted 5+ times
Frontend — MemoriesTab.jsx, TrackTab.jsx (feeding, sleep, poop), HealthTab.jsx (growth records)
The "Delete? Yes / No" confirmation pattern — with confirmDeleteId state, conditional rendering — appears in at least 5 places with nearly identical JSX. Any design change to the confirmation UI requires 5 edits.
Fix: Create a <ConfirmDelete onConfirm={fn} onCancel={fn} /> component and replace all instances. The state management can stay in the parent.

P3

Backend/src/main/java/com/gotcherapp/api/baby/ — BabyProfileResponse.java, BabyProfileRequest.java
The baby profile DTO uses babyName (camelCase) which Jackson serializes to babyName in JSON. The frontend then uses profile.babyName. All other DTOs use snake_case field names that Jackson serializes directly (e.g. display_name, email_verified). Fix: Decide on one convention. Either add @JsonNaming(SnakeCaseStrategy.class) globally, or keep camelCase and document it. The frontend already handles both; the important thing is consistency going forward.

Documentation Gaps

Places where adding documentation would most help a future engineer or AI assistant understand the codebase. High = will cause confusion or incorrect behavior without it; Med = slows understanding; Low = nice to have.

High
api.js — 401 retry and session-expired dispatch are invisible to callers
Frontend/src/lib/api.js
The module-level refreshing Promise that deduplicates concurrent 401s, the session-expired custom event dispatch, and the fact that apiRequest will retry once after a token refresh are all non-obvious side effects with no JSDoc. A future developer adding a new API call has no way to know that 401 responses are transparently retried, or that a failed refresh fires a global event that boots the user.
Add: JSDoc on apiRequest documenting: parameters, return value, 401-retry behaviour, and the session-expired event. Add a one-line comment above let refreshing = null explaining it deduplicates concurrent refreshes.
High
pdf.js — generatePdf entry shape is undocumented
Frontend/src/lib/pdf.js
generatePdf(entries, babyName) requires each entry to have specific fields: week, title, story, image_url, image_orientation, and entry_date (or date). None of this is documented. The entry_date || entry.date fallback pattern (lines 202, 235) hints at a legacy field rename that could silently break if a caller uses the wrong field name.
Add: JSDoc on generatePdf listing the expected shape of each entry object. Also clarify the generatePdf vs downloadPdf paywall seam — the current comment ("paywall seam between the two") is minimal.
Med
babyAge.js — getMilestones key format undocumented
Frontend/src/lib/babyAge.js · getMilestones()
The milestone key format ${groupWeek}-${index} (where groupWeek = Math.floor(week/4)*4) is the most non-obvious invariant in the frontend — using the wrong format was an actual regression in a prior session. It is only documented in seed-demo-user.sh comments, not in getMilestones, DashboardTab, or HealthTab where keys are constructed.
Add: A one-line comment above getMilestones explaining the key format and the 4-week bucket grouping. Same comment at each call site that constructs a key.
Med
AuthService.java — token rotation strategy undocumented
Backend/src/main/java/com/gotcherapp/api/auth/AuthService.java · refresh()
The refresh() method revokes the old refresh token and issues a new pair — a security pattern that prevents replay attacks. There is no comment explaining why the old token is revoked or what threat model this defends against. The is_revoked column in refresh_tokens exists solely because of this rotation pattern, which is also not documented in the migration.
Add: A one-line comment above the UPDATE ... SET is_revoked = TRUE line: "// Rotate: revoke old token to prevent replay — each refresh token is single-use."
Med
README — missing test and seed commands
README.md (root)
The README has no "How to run tests" section. A developer joining the project cannot discover that cd Frontend && npm test runs Vitest or that cd Backend && ./gradlew test runs JUnit without reading source files. There is also no mention of ./seed-demo-user.sh for test data or how to add a new Flyway migration.
Add: A "Development" section to README.md covering: run frontend tests (cd Frontend && npm test), run backend tests (cd Backend && ./gradlew test), seed demo data (./seed-demo-user.sh), and add a migration (db/migration/V{N}__description.sql).
Med
share.js — return value contract undocumented
Frontend/src/lib/share.js
shareFirstTime() returns 'copied' when the clipboard fallback is used, and undefined (implicitly) when Web Share is used. Callers in FirstTimeCard.handleShare check for this return value to show a "Copied!" toast. This asymmetric return contract is invisible without reading both files.
Add: A JSDoc comment: /** @returns {'copied' | undefined} 'copied' if clipboard fallback was used, undefined if Web Share API handled it */
Med
application.properties — SECURE_COOKIES env var undocumented
Backend/src/main/resources/application.properties
The SECURE_COOKIES environment variable is consumed by CookieUtil.java via System.getenv, bypassing Spring's property system. It does not appear in application.properties or .env.example. A deployer setting up production will miss it entirely unless they read the source.
Add: app.cookies.secure=${SECURE_COOKIES:false} to application.properties with a comment: "# Set SECURE_COOKIES=true in production (requires HTTPS)". Also add SECURE_COOKIES=true to .env.example.
Low
dashboardStats.js — input shape and return contract undocumented
Frontend/src/lib/dashboardStats.js
computeFeedingStats(feeding) and computeSleepStats(sleep) accept arrays of log objects but have no doc on what fields are expected (startedAt, endedAt) or what each returned stat means (mostRecentFeed, avgGapStr, completedSleep, etc.).
Add: Brief JSDoc on both exported functions listing input shape and return fields. These are tested, so the doc can reference the test file for examples.
Low
formatting.js — entry_date vs date field duality undocumented
Frontend/src/lib/formatting.js · formatEntryDate()
formatEntryDate(date) accepts either a Date object or a string but callers in pdf.js pass entry.entry_date || entry.date, relying on both field names being tried. The function itself is fine but the dual-field pattern is a hidden contract that callers must know about.
Add: A one-line comment in pdf.js noting why entry_date || entry.date is used — legacy field rename from when journal entries switched field names.
Med
Backend db/migration — V7 out-of-order flag context is buried
Backend/src/main/resources/application.properties · line 16
The comment "V7 was added as a placeholder after V8 was already applied" in application.properties explains the out-of-order=true flag. But the context for WHY V7 is a placeholder is not in the migration file itself, making it hard to understand when reviewing db/migration/ in isolation. A new dev might delete the flag thinking it's no longer needed.
Add: A comment at the top of V7__*.sql explaining it was inserted after V8 and requires spring.flyway.out-of-order=true. This makes the migration directory self-documenting.
